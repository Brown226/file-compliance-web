## 一、Python 文件解析微服务

**状态**: 🟢 已完成

**问题**: 当前 Node.js 的文件解析库（mammoth / pdf-parse / xlsx）对中文 PDF 提取质量差，DWG 无成熟方案。

**已实施方案**:
1. 创建 Python FastAPI 微服务项目 `file-parser-service/`
2. PDF 解析：PyMuPDF (fitz) + pdfplumber 双引擎，中文提取准确性大幅提升 + 表格结构提取
3. DWG/DXF 解析：ezdxf 读取图层信息、文本实体、尺寸标注
4. Word 解析：python-docx，保留段落样式和表格结构
5. Excel 解析：openpyxl，保留 sheet 结构
6. 统一 API：`POST /api/parse` 返回 `{text, pages, metadata, structure}` 结构化数据
7. Node.js 后端集成：`PythonParserService` 客户端 + `ParserService` 优先调用 Python 服务 + Node.js fallback
8. DWG 文件审查增强：解析成功时提取 dimensions 数据并关联 cadHandleId
9. Docker Compose 集成：新增 `file-parser` 服务（:8000）
10. 环境变量：`PARSER_SERVICE_URL=http://localhost:8000`

**待确认项**:
- [ ] 是否需要 Word/Excel 也换成 Python 解析？还是只换 PDF + DWG？
- [ ] Python 微服务的部署方式：独立 Docker 容器？还是与现有 docker-compose 合并？
- [ ] 是否需要同时保留 Node.js 解析作为 fallback？
- [ ] 解析结果是否需要返回结构化数据（段落/表格/样式），还是纯文本即可？

**确认内容**:
    所有服务全部采用python库进行解析，部署方式选择与当前docekr-compose合并，解析结果需要返回结构化数据（段落/表格/样式）。

## 四、批量同步实时进度反馈

**状态**: 🟢 已完成

**问题**: 批量同步标准到 MaxKB 时无实时进度反馈，用户不知道执行到哪一步。

**已实施方案**:
1. 后端新增 SSE 流式同步 API: `POST /api/maxkb/sync/all-standards-stream`
2. MaxKBService 新增 `syncAllStandardsWithProgress()` 方法，支持进度回调
3. SSE 事件类型: `progress`(逐条进度)、`done`(全部完成)、`error`(异常)
4. 前端使用 `fetch + ReadableStream` 实现（保留 Bearer token 认证）
5. 前端 LLMConfig.vue 新增同步进度弹窗（进度条 + 成功/失败计数 + 逐条日志）

**待确认项**:
- [ ] 使用 SSE 还是 WebSocket？（SSE 更轻量，单向推送足够）
- [ ] 前端进度展示形式：弹窗？侧边栏？页面内嵌？

**确认内容**:
    使用 SSE 推送进度
    前端进度展示形式：弹窗

- [ ] 是否需要支持 DWG 文件的预览/渲染？

    不需要支持 DWG 文件的预览/渲染
