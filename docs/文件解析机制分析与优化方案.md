## 一、当前系统文件解析机制详解

### 1.1 整体架构

当前系统采用 **Python 优先 + Node.js Fallback** 的双引擎解析架构：

```
┌──────────────────────────────────────────────────────────────────┐
│                    ParserService.parseFile()                     │
│                                                                  │
│  ① Python FastAPI 微服务 (优先)                                  │
│     POST http://localhost:8000/api/parse                         │
│     → 返回 ParseResult 结构化数据                                │
│     → 仅取 .text 纯文本字符串返回                                │
│                                                                  │
│  ② Node.js 本地解析 (Fallback)                                  │
│     docx → mammoth.extractRawText()  → 纯文本                   │
│     xlsx → XLSX 逐 sheet 逐行        → "cell|cell" 管道分隔     │
│     pdf  → pdf-parse                 → 纯文本                   │
│     dwg  → 占位返回空字符串                                      │
│                                                                  │
│  ③ OCR 降级 (条件触发)                                           │
│     PDF 文本<20字符 → PaddleOCR-VL base64 识别                   │
│     DWG 解析失败     → PaddleOCR-VL base64 识别                  │
└──────────────────────────────────────────────────────────────────┘
```

### 1.2 各文件类型解析输出格式

| 文件类型 | Python 解析输出 | Node.js Fallback | 输出格式 |
|---------|---------------|------------------|---------|
| **docx/doc** | 结构化 `ParseResult`（含 paragraphs/tables/headers） | `mammoth.extractRawText()` 提取纯文本 | 纯字符串，`\n` 分隔 |
| **xlsx/xls** | 结构化 `ParseResult` | XLSX 逐 sheet 逐行提取 | `cell1 \| cell2 \| cell3` |
| **pdf** | 结构化 `ParseResult`（含 `pages[]`） | `pdf-parse` 全文纯文本 | 纯字符串 |
| **dwg** | 结构化 `ParseResult`（含 dimensions/layers/handles） | 空字符串（完全依赖 Python） | 纯字符串或空 |

### 1.3 Python 解析服务的完整输出结构

```typescript
interface ParseResult {
  text: string;              // 纯文本（用于 LLM 审查）
  pages: string[];           // 逐页文本
  metadata: {
    page_count?: number;
    has_tables: boolean;
    has_images: boolean;
    dwg_layers?: string[];
    dwg_text_count?: number;
    dwg_dimension_count?: number;
    parse_error?: string;
  };
  structure: {
    paragraphs: Array<{
      text: string;
      style: string | null;   // Heading1, Normal 等
      page: number | null;
      handle?: string;        // DWG 实体句柄
    }>;
    tables: Array<{
      page: number | null;
      rows: string[][];
      caption: string | null;
    }>;
    dimensions?: Array<{      // DWG 标注
      text: string;
      layer: string;
      entity_type: string;
      handle: string;
      measurement?: string | null;
    }>;
  };
}
```

### 1.4 OCR 处理机制

当前使用 **硅基流动 PaddleOCR-VL-1.5** 模型：

| 属性 | 值 |
|------|-----|
| 模型 | PaddlePaddle/PaddleOCR-VL-1.5 |
| API 格式 | OpenAI Chat Completions 兼容 |
| 调用方式 | 文件 → base64 → Data URL → 多模态消息 |
| 触发条件 | PDF 文本 < 20 字符 / DWG 解析失败 |
| 超时 | 180s |
| 后处理 | 字符纠正（长破折号→连字符、全角→半角） |
| 支持格式 | pdf, png, jpg, jpeg, gif, webp, bmp, tiff |

### 1.5 Word 结构提取

`WordStructureService` 采用两级策略：

1. **优先**：使用 Python 解析结果的 `structure` 字段
2. **Fallback**：使用 `jszip` 解析 docx XML（`word/document.xml`、`word/header1.xml`）
3. **最终降级**：`ParserService.parseFile()` 拆行

输出结构：
```typescript
interface WordStructure {
  paragraphs: Array<{ text: string; style: string | null; page: number | null }>;
  headers: Array<{ text: string; type: 'odd' | 'even' | 'first'; page: number | null }>;
  tables: Array<{ rows: string[][]; caption: string | null }>;
}
```

### 1.6 传递给 LLM 的数据流

```
文件 → ParserService.parseFile() → 纯文本 extractedText
                                    ↓
                              LlmService.splitText()
                              (4000 字符/片，段落优先→句子回退)
                                    ↓
                              组装 Prompt:
                                system: 场景化审查 prompt
                                user: 【审查标准】+ 【待审查文本】
                                    ↓
                              OpenAI 兼容 API (/chat/completions)
                                    ↓
                              parseReviewResult() → ReviewIssue[]
```

### 1.7 核心矛盾

| 问题 | 影响 |
|------|------|
| **纯文本丢结构** | LLM 无法区分标题、正文、表格、页眉，审查精度受限 |
| **表格扁平化** | `cell\|cell\|cell` 格式丢失列头，LLM 难以理解表格语义 |
| **样式信息全丢** | Heading1/加粗/列表等格式线索被丢弃，无法做格式合规审查 |
| **分片不感知结构** | 4000字符硬切可能切断表格/列表，LLM 看到残缺内容 |
| **前端预览差** | 纯文本展示无格式，可读性极低 |

## 二、markitdown 技术原理与适用性评估

### 2.1 项目概览

| 属性 | 值 |
|------|-----|
| 开发者 | Microsoft AutoGen 团队 |
| 许可证 | MIT |
| 语言 | Python 3.10+ |
| GitHub Stars | 112k |
| 最新版本 | v0.1.5 (2026-02) |
| 定位 | 将文件转为 Markdown 供 LLM 消费的轻量级工具 |

### 2.2 支持的格式

| 格式 | markitdown | 当前系统 | 差异 |
|------|-----------|---------|------|
| DOCX | ✅ 标题/表格/列表 → Markdown | ✅ mammoth 纯文本 | markitdown 保留结构 |
| XLSX | ✅ 表格 → Markdown table | ✅ `cell\|cell` 管道分隔 | markitdown 保留列头 |
| PDF | ✅ 文本+表格 → Markdown | ✅ pdf-parse 纯文本 | markitdown 表格识别更好 |
| PPTX | ✅ 幻灯片内容 → Markdown | ❌ 不支持 | 新增支持 |
| HTML | ✅ 转 Markdown | ❌ 不支持 | 新增支持 |
| 图片 | ✅ EXIF + LLM 视觉描述 | ❌ 仅 OCR | 语义更丰富 |
| 音频 | ✅ 语音转录 | ❌ 不支持 | 新增支持 |
| DWG | ❌ 不支持 | ✅ Python 微服务 | 当前方案优势 |

### 2.3 内部转换实现

| 文件类型 | 底层库 | 输出 Markdown 格式 |
|---------|--------|-------------------|
| DOCX | `python-docx` | `# 标题` / `\| 表头 \| 表头 \|` / `- 列表项` |
| XLSX | `openpyxl` | `\| 列1 \| 列2 \|` Markdown 表格 |
| PDF | `pdfminer.six` | 段落 + 表格（Markdown 表格语法） |
| PPTX | `python-pptx` | `## 幻灯片 N` + 内容 |
| 图片 | PIL + LLM | `![描述](path)` 或文本描述 |
| HTML | `beautifulsoup4` | 标准 Markdown |

### 2.4 集成方案

**推荐方案**：在 Python 微服务内部替换解析引擎

```python
from markitdown import MarkItDown
md = MarkItDown(enable_plugins=False)

@app.post("/api/parse")
async def parse_file(file: UploadFile, file_type: str):
    content = await file.read()
    result = md.convert_stream(io.BytesIO(content), file_extension=f".{file_type}")
    markdown_text = result.text_content
    # 同时提供结构化数据（兼容规则引擎）
    parse_result = {
        "text": markdown_text,
        "markdown": markdown_text,
        "pages": [...],
        "structure": {...},
    }
    return {"code": 200, "data": parse_result}
```

### 2.5 预期收益

| 收益 | 量化预估 |
|------|---------|
| LLM 审查精度 | 提升 20-30%（表格/标题结构可识别） |
| 前端原文预览 | 从纯文本 → Markdown 渲染 |
| Excel 审查能力 | `cell\|cell` → Markdown 表格，列头可识别 |
| PPTX 支持 | 新增能力 |
| Token 效率 | Markdown 比 JSON 结构节省 30-50% Token |
| 维护成本 | 减少自研 Python 解析逻辑 |

### 2.6 风险与注意事项

| 风险 | 缓解措施 |
|------|---------|
| 中文 PDF 表格识别 | 需实际测试，可能需要 Azure Document Intelligence 补强 |
| DWG 不支持 | 保留现有 Python DWG 解析逻辑 |
| 大文件性能 | markitdown 流式处理，但超大 PDF 可能较慢 |
| OCR 对比 | markitdown-ocr 用 LLM Vision，当前用 PaddleOCR-VL，需对比 |

## 三、OCR 服务对比分析

> 详见本文档第四章"替代方案对比"

### 当前方案：PaddleOCR-VL-1.5（硅基流动托管）

**优势**：
- OpenAI 兼容 API，集成简单
- 中文识别能力强
- 支持图文混排

**劣势**：
- 依赖第三方云服务（硅基流动）
- 大文件 base64 编码传输效率低
- 无法本地部署（内网环境受限）
