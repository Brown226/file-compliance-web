## 二、各改进点详细方案及多智能体协作

### P0-1：路由管理员守卫缺失

**问题**：`router/index.ts` 的 `beforeEach` 只检查 `requiresAuth`，未检查 `requiresAdmin`，`/pipeline-config`、`/prompt-config` 等路由可通过直接 URL 访问。

**修复方案**：
```ts
// router/index.ts beforeEach 中添加
if (to.meta.requiresAdmin && userStore.userInfo?.role !== 'ADMIN') {
  next({ path: '/dashboard' })
  return
}
```

**多智能体协作**：

| 角色 | 分工 | 交付物 |
|------|------|--------|
| 前端开发 | 修改 router beforeEach 添加 requiresAdmin 检查 | 代码变更 |
| 测试 | 验证非 ADMIN 用户直接访问 /pipeline-config 被拦截；ADMIN 用户正常访问 | 测试报告 |
| 代码审查 | 检查是否有其他路由缺少守卫、meta 定义是否完整 | 审查意见 |

**执行流程**：前端开发修改 → 代码审查 → 测试验证 → 合并

**问题**：`main.ts` 全量导入 Element Plus 和图标。

**修复方案**：

| 角色 | 分工 | 交付物 |

### P1-5：类型安全加固

**问题**：`role` 类型不一致、API 层大量 `any`、API 响应无类型。

**修复方案**：
1. 统一 `role` 类型为 `'ADMIN' | 'MANAGER' | 'USER'`
2. 创建 `types/api.d.ts` 定义所有 API 响应类型
3. 创建 `types/models.d.ts` 定义业务模型类型
4. 逐步消除 `api/` 目录中的 `any` 类型

**多智能体协作**：

| 角色 | 分工 | 交付物 |
|------|------|--------|
| 前端开发 | 定义类型文件，修改 store 和 API 层 | 类型定义 + 代码变更 |

| 代码审查 | 检查类型完整性、是否有遗漏的 any | 审查意见 |

**交付标准**：`api/` 目录零 `any`；store 类型一致；`types/` 目录有完整导出

## 三、新增组件集成方案

### 已采纳组件

| 组件 | 优先级 | 集成阶段 | 用途 |
|------|--------|----------|------|
| `@vueuse/core` | P0 | Phase 0 | Composables 基础，提供 `useDebounceFn`、`useResizeObserver`、`useStorage` 等 |
| `@vue-office/docx` + `@vue-office/pdf` | P1 | Phase 5 | TaskDetails 文档预览（组件拆分后集成） |
| `markdown-it` + `dompurify` | P1 | Phase 5 | LLM 审查结果 Markdown 渲染（含 XSS 防护） |

### 降级/替代方案

| 组件 | 决定 | 理由 |
|------|------|------|
| `vxe-table` | **拒绝**，用 `el-table-v2` | 数据量不超 10000 行，`el-table-v2` 虚拟滚动已够用，避免引入新 UI 框架 |
| Monaco Editor | **推迟**，考虑 CodeMirror 6 | Monaco 3-4MB 太重，CodeMirror 6 仅 200-400KB 更适合内网系统 |
| Uppy | **降级 P3** | 用 `@vueuse/core` 的 `useFileDialog` + `useDropZone` 增强即可 |
| `vuedraggable` | **降级 P3** | 当前无拖拽排序刚需，仅在知识库管理场景考虑 |

### 集成注意事项

1. **@vue-office/docx+pdf**：必须在 TaskDetails 组件拆分后集成，预览组件作为 `DocumentPreview.vue` 子组件
2. **markdown-it+dompurify**：必须搭配 DOMPurify 防止 XSS，LLM 输出不可信
3. **@vueuse/core**：作为 Composables 提取的基础，Phase 0 安装后逐步替代手写逻辑

## 四、总体执行策略

### 分阶段执行

```
Phase 0（立即）：P0 安全修复 + 安装 @vueuse/core
Phase 1（第1周）：P1 性能优化（Element Plus + ECharts 按需导入）
Phase 2（第2周）：P1 Composables 提取（基于 @vueuse/core）
Phase 3（第3周）：P1 巨型组件拆分（StandardLibrary + TaskDetails）
Phase 4（第4周）：P1 巨型组件拆分（NewTask + LLMConfig）+ 类型安全
Phase 5（第5周）：新组件集成（@vue-office/docx+pdf, markdown-it+dompurify）
Phase 6（第6周）：P2 架构优化（Vite 构建、SSE 规范化、CSS 变量统一等）
Phase 7（第7周）：P3 打磨 + 全面回归测试
```

### 智能体团队配置

```
Team: frontend-optimization
├── frontend-dev    → 负责代码实现、组件拆分、composables 提取、组件集成
├── code-reviewer   → 负责代码质量、架构合规、安全审计
├── ui-designer     → 负责 CSS 变量、交互规范、视觉一致性
└── tester          → 负责功能验证、回归测试、性能基准
```

### 协作流程（每个改进点）

```
1. [frontend-dev] 实现变更
2. [code-reviewer] 审查代码质量 + 安全
3. [tester] 功能回归 + 性能对比
4. [ui-designer] 视觉/交互一致性确认（视需要）
5. [frontend-dev] 修复审查/测试反馈
6. 合并
```

### 交付标准总则

- 每个改进点完成后，相关页面功能零退化
- P1 改进需提供性能对比数据（bundle 体积、加载时间）
- 所有新代码有 TypeScript 类型，无 `any`
- Composables 和新组件有 JSDoc 注释
- 安全修复需有测试用例覆盖
- 新组件集成需有独立可运行的 demo 验证
