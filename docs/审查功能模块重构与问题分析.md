## 一、现状分析：审查任务入口单一导致逻辑交叉

### 1.1 问题概述

> **📋 状态更新 (2026-04-15)**：本节描述的"单一入口+全量流水线"架构问题已在 P0/P1 阶段完成解耦。具体变更见文末"七、P0/P1 实施完成记录"。

当前系统仅有一个审查任务创建入口 — `NewTask.vue` → `POST /api/tasks`。所有审查需求（无论类型）都通过同一个入口提交，后端 `ReviewService.processFile()` 以固定流水线处理所有文件：

```
文件解析 → OCR降级 → PDF逐页提取 → 规则引擎(10类全量) → 标准引用检查 → MaxKB审查 → 写结果
```

**核心问题**：无论用户想要执行哪种审查（以库审文 / 以文审文 / 错别字检查 / 一致性检查 / 多模态识别），系统都会执行完整的流水线，导致：

1. **资源浪费**：用户只想检查错别字，却触发了 MaxKB 审查和标准引用比对
2. **结果混淆**：不同类型的审查结果混杂在同一个 `task_details` 表中，用户难以区分
3. **配置矛盾**：MaxKB 审查需要标准库已同步，但错别字检查完全不需要标准库
4. **前端限制**：`NewTask.vue` 的"审查标准"必选项暗示所有任务都需要标准，但实际很多审查类型不需要

### 1.2 代码层面的问题定位

#### 1.2.1 前端：单一表单，单一入口

```typescript
// NewTask.vue - 表单定义
const form = reactive({
  title: '',
  standardIds: [] as string[],  // 所有任务都要选标准？
  description: ''
});
```

- 表单中 `standardIds` 没有区分审查类型
- 没有审查模式/类型的选项
- 上传文件后统一调用 `createTaskApi(formData)`，无法传递审查意图

#### 1.2.2 后端：Task 模型缺少审查类型字段

```prisma
// schema.prisma - Task 模型
model Task {
  id            String         @id @default(uuid())
  title         String
  description   String?
  status        TaskStatus     @default(PENDING)
  creatorId     String
  standardId    String?        // 单一标准关联
  standard      Standard?      @relation(...)
  files         TaskFile[]
  details       TaskDetail[]
  taskStandards TaskStandard[]
  // ❌ 缺少: reviewType / reviewMode 字段
}
```

- Task 模型没有 `reviewType` 字段来区分审查模式
- `standardId` 是可选的，但 `NewTask.vue` 暗示它是必要的
- 无法在数据库层面追溯某个任务的审查类型

#### 1.2.3 ReviewService：全量流水线，无条件分支

```typescript
// review.service.ts - processFile() 流水线
// 步骤 1: 文件解析（所有类型都执行）
// 步骤 2: OCR 降级（所有类型都执行）
// 步骤 3: PDF 逐页提取（所有类型都执行）
// 步骤 4: 规则引擎（10类规则全量执行，无条件跳过）
// 步骤 5: 标准引用检查（所有类型都执行）
// 步骤 6: DWG 特殊处理（仅 DWG，但逻辑嵌在主流程中）
// 步骤 7: 无文本跳过（所有类型都执行）
// 步骤 8: MaxKB 审查（所有类型都执行 — 最大浪费点）
```

- `RuleEngineService.runAllRules()` 执行全部 10 类规则，无法按需选择
- MaxKB 审查是唯一的 AI 引擎，必须检查初始化状态、健康状态才能执行
- 标准引用检查 `checkStandardReferences()` 无条件执行，即使用户不需要

### 1.3 问题影响矩阵

| 影响维度 | 严重程度 | 描述 |
|---------|---------|------|
| 性能 | 🔴 高 | 全量流水线执行，MaxKB 调用成本高（每次分片审查都需 API 调用） |
| 用户体验 | 🔴 高 | 结果混杂，无法按审查意图过滤；强制选标准库令人困惑 |
| 可维护性 | 🟡 中 | processFile 方法逻辑过重（280+行），新增审查类型需修改主线代码 |
| 可扩展性 | 🔴 高 | 新增审查模式只能在现有流水线中硬编码分支，违反开闭原则 |
| 成本 | 🟡 中 | 无谓的 MaxKB/OCR 调用产生不必要的 API 费用 |

## 二、功能模块重新梳理与分类设计

### 2.1 审查模式分类

基于业务场景，将审查功能拆分为 **6 大独立审查模式**，每种模式有独立的触发条件、流水线编排和结果格式：

| 审查模式 | 代号 | 核心能力 | 必要输入 | 可选输入 |
|---------|------|---------|---------|---------|
| **以库审文** | `LIBRARY_REVIEW` | 依据法规标准库审查下游文件全要素符合性 | 待审文件 + 标准库 | 审查范围（全要素/指定要素） |
| **以文审文** | `DOC_REVIEW` | 依据上游文件批量化自动化审查下游文件 | 上游参照文件 + 下游待审文件 | 映射规则 |
| **全文一致性** | `CONSISTENCY_CHECK` | 检测文档内部上下文矛盾及修改未同步 | 待审文件 | 无 |
| **错别字/语法** | `TYPO_GRAMMAR` | AI 语义一键式检查 | 待审文件 | 检查级别（严格/普通） |
| **多模态识别** | `MULTIMODAL_OCR` | 文字、表格、复杂公式自动识别审查及生成报告 | 待审文件（图片/PDF） | 输出格式 |
| **自定义规则审查** | `CUSTOM_RULE_REVIEW` | 用户自选启用规则+自定义参数的确定性规则审查 | 待审文件 | 规则选择、参数配置 |

> **关键定位**：自定义规则审查是 **横切能力**，不是独立的审查模式。它与其他 5 种模式的关系是 **"配置层 vs 执行层"**：
> - 自定义规则（启用/禁用/参数调整）影响的是 **所有模式中规则引擎步骤的行为**
> - 以库审文、全文一致性等模式决定了 **走哪些流水线步骤**（OCR/MaxKB/标准引用等）
> - 自定义规则决定了 **规则引擎步骤内部执行哪些规则、用什么参数**
> - 因此：`CUSTOM_RULE_REVIEW` 可以作为独立入口（用户只想跑确定性规则，不调 AI），也可以作为其他模式的配置项

## 三、五大新增功能深度分析

### 3.1 以库审文（依据法规标准库审查下游文件全要素符合性）

#### 3.1.1 当前实现状态

**已实现部分**：
- 标准库管理（CRUD + 文件夹分类 + Excel导入导出）
- MaxKB 知识库同步（单条同步/批量同步/自动重同步）
- MaxKB RAG 审查（`MaxKBService.reviewViaMaxKB()`）
- 标准引用规范性检查（`StandardExtractorService` + `StandardCheckService` + `CharDiffService`）
- 多标准关联（`taskStandards` 多对多关系）

**未实现部分**：
- 审查范围选择（全要素/指定要素）
- 审查结果与标准条文的精确关联映射
- 审查覆盖率统计（已覆盖标准条文/总条文数）

#### 3.1.2 存在的问题

| # | 问题 | 严重程度 | 详细说明 |
|---|------|---------|---------|
| 1 | **MaxKB 依赖过强** | 🔴 高 | 当前 MaxKB 是唯一 AI 引擎，若 MaxKB 服务不可用，整个审查流程直接抛异常终止（`review.service.ts:224-236`）。没有 fallback 到直接 LLM 调用的降级方案 |
| 2 | **RAG 检索质量不可控** | 🔴 高 | MaxKB 的 RAG 检索质量取决于：知识库分片策略、embedding 模型、检索 top-k 参数，这些参数在当前系统中不可配置。审查结果可能遗漏关键标准条文 |
| 3 | **标准同步状态与审查结果不一致** | 🟡 中 | 标准更新后 `maxkbSyncStatus` 标记为 `pending`，自动重同步是异步的。如果用户在同步完成前发起审查，MaxKB 使用的仍是旧版标准内容 |
| 4 | **审查结果缺少标准条文溯源** | 🟡 中 | MaxKB 返回的审查结果只有 `issueType/originalText/suggestedText/description`，无法追溯具体违反了哪个标准的哪条条文。`ruleCode` 字段在 LLM 结果中始终为 `null` |
| 5 | **临时标准库重启丢失** | 🟡 中 | `StandardService.tempLibraryCache` 使用内存 Map 存储，服务重启后丢失。对于重要审查任务，用户需要重新导入 |
| 6 | **多标准审查无优先级** | 🟢 低 | 多标准关联 (`taskStandards`) 是平等的，无法设置主标准/辅助标准的优先级关系 |

#### 3.1.3 改进建议

1. **增加 LLM 直接调用降级**：当 MaxKB 不可用时，回退到将标准内容拼入 prompt 直接调用 LLM
2. **审查结果增加标准条文关联**：在 MaxKB prompt 中要求返回违反的标准编号和条文号，映射到 `ruleCode`
3. **同步状态检查前置**：审查启动前检查所有关联标准的同步状态，未完成时提示用户
4. **临时标准库持久化**：使用 Redis 或数据库替代内存 Map

### 3.2 以文审文（依据上游文件批量化自动化审查）

#### 3.2.1 当前实现状态

**🔴 完全未实现**

当前系统没有任何"以文审文"功能的支持：
- Task 模型没有上游参照文件的概念
- ReviewService 没有文件间比对逻辑
- 前端没有上传参照文件的入口
- 没有"上游-下游映射规则"的配置

#### 3.2.2 需要新增的能力

| # | 能力 | 说明 |
|---|------|------|
| 1 | **上游参照文件组** | 新增 `RefFileGroup` + `RefFile` 模型，存储上游参照文件 |
| 2 | **文件解析（参照组）** | 对上游文件执行同样的解析+OCR流程 |
| 3 | **上下文比对引擎** | 提取上游文件的关键参数/数据/结论，与下游文件对比 |
| 4 | **LLM 辅助比对** | 将上游文件内容作为参照上下文，发送给 MaxKB/LLM 审查下游文件 |
| 5 | **映射规则配置** | 允许用户定义上游-下游的对应关系（如：上游"设计参数"→下游"施工参数"） |
| 6 | **差异高亮** | 前端展示上游/下游不一致的具体差异位置 |

#### 3.2.3 技术方案

```typescript
// 以文审文核心流程
class DocReviewPipeline {
  async execute(taskId: string, file: TaskFile, ctx: FileContext): Promise<RuleIssue[]> {
    // 1. 获取上游参照文件组
    const refGroup = await prisma.refFileGroup.findUnique({
      where: { taskId },
      include: { files: true },
    });
    
    // 2. 解析所有上游参照文件
    const refTexts = await Promise.all(
      refGroup.files.map(f => ParserService.parseFile(f.filePath, f.fileType))
    );
    
    // 3. 构建参照上下文
    const refContext = refTexts.join('\n---\n');
    
    // 4. 通过 MaxKB/LLM 比对审查
    const prompt = `以下为上游参照文件内容:\n${refContext}\n\n请审查以下下游文件是否与上游文件一致...`;
    const issues = await MaxKBService.reviewWithReference(prompt, ctx.extractedText);
    
    return issues;
  }
}
```

#### 3.2.4 关键风险

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 上游文件解析失败导致参照缺失 | 审查结果不完整 | 上游文件解析失败时中止任务并提示 |
| 参照上下文过长超出 LLM 窗口 | 比对不完整 | 智能摘要上游文件 + 分段比对 |
| 上游-下游映射关系需要人工定义 | 自动化程度受限 | 提供默认映射模板 + 用户自定义 |

### 3.3 全文一致性（检测上下文矛盾及修改未同步）

#### 3.3.1 当前实现状态

**部分实现**：
- 规则引擎中有 `consistency.rule.ts`，实现了：
  - `CONSIST_001`：项目名称一致性（封面 vs 正式名称）
  - `CONSIST_002`：目录编码与正文编码一致性
- 规则引擎中有 `encoding.rule.ts`，实现了：
  - `CODE_001`：页眉编码与文件名编码不一致
  - `CODE_002`：页眉使用了内部编码
  - `UNIT_001`：机组号不一致

**未实现部分**：
- **跨页一致性检查**：同一参数在不同页面出现不同值
- **修改未同步检测**：某处修改后，引用该参数的其他位置未同步更新
- **图表与正文一致性**：表格/图表中的数据与正文描述不一致
- **LLM 辅助深层一致性检查**：规则引擎只能检测格式层面的不一致，语义层面的矛盾需要 LLM

#### 3.3.2 存在的问题

| # | 问题 | 严重程度 | 详细说明 |
|---|------|---------|---------|
| 1 | **一致性检查仅限格式层面** | 🔴 高 | 当前一致性规则只能检测编码/项目名的格式不一致，无法检测语义矛盾（如"设计温度350°C" vs "设计温度360°C"） |
| 2 | **无跨文件一致性检查** | 🔴 高 | 当前 `processFile()` 是单文件独立处理，无法跨文件比较。同一个项目下多个文件间的一致性问题完全无法检测 |
| 3 | **规则覆盖率极低** | 🟡 中 | `consistency.rule.ts` 仅 2 条规则（`CONSIST_001`/`CONSIST_002`），远不能覆盖工程文件中常见的一致性问题类型 |
| 4 | **缺少"修改未同步"检测机制** | 🔴 高 | 工程文件中最常见的一致性问题：A 处修改了参数，但 B 处/C 处未同步更新。当前系统完全没有此能力 |
| 5 | **MaxKB 用于一致性检查的效果未验证** | 🟡 中 | 一致性检查需要 LLM "记住"全文上下文并检测矛盾，但当前 MaxKB 审查是分片进行的（4000字/片），跨片矛盾可能漏检 |

#### 3.3.3 改进建议

1. **引入参数抽取与索引机制**：从全文中抽取所有"参数名=参数值"对，建立参数索引表，检测同名参数的不同取值
2. **跨文件一致性检查**：在 `processTask()` 层面（而非 `processFile()` 层面）增加全局一致性比对步骤
3. **LLM 全文一致性审查**：将全文摘要发送给 LLM，专门检测语义矛盾（需控制 token 消耗）
4. **修改未同步检测**：对比不同版本文件，识别变更点并检查所有引用点是否同步

#### 3.4.2 存在的问题

|---|------|---------|---------|
| 1 | **规则引擎的 typo 检查名不副实** | 🔴 高 | `typo.rule.ts` 实际只检查编码错误（`TYPO_001`），不是错别字检查。文件名与实际功能不匹配，容易误导开发者 |

5. **误报反馈机制**：在 TaskDetail 中增加 `isFalsePositive` 字段，用户标记后纳入统计

### 3.5 多模态识别（文字、表格、复杂公式自动识别审查及生成报告）

#### 3.5.1 当前实现状态

**部分实现**：
- `OcrService`：调用硅基流动 PaddleOCR-VL-1.5 进行图片/PDF 文字识别
- OCR 后处理：字符纠正（破折号/中文"一"→连字符）

**未实现部分**：

- **复杂公式识别**：PaddleOCR-VL 对数学公式的识别能力有限，且无法输出 LaTeX/MathML 格式

- **结构化报告生成**：没有基于识别结果自动生成审查报告的功能
- **图表审查**：无法对流程图、系统图等图形化内容进行合规性检查

#### 3.5.2 存在的问题

| # | 问题 | 严重程度 | 详细说明 |
|---|------|---------|---------|
| 1 | **OCR 仅用于文本提取降级** | 🔴 高 | 当前 OCR 是作为"文本解析失败时的降级方案"（`ParserService.needsOcr()` 判断），而不是作为多模态识别的核心能力。OCR 结果只是纯文本字符串，丢失了所有结构信息 |
| 2 | **表格识别能力缺失** | 🔴 高 | Python 微服务中 pdfplumber 可提取表格，但 `ParserService.parseFile()` 只返回纯文本。表格数据在审查流程中完全丢失结构 |
| 3 | **公式识别完全未实现** | 🔴 高 | PaddleOCR-VL 不是专门的公式识别模型。核电工程文件中的公式（如安全系数计算公式）需要专门的数学公式 OCR 模型（如 LaTeX-OCR/Pix2Tex） |
| 4 | **多模态审查逻辑空白** | 🔴 高 | OCR 识别后的文字只进入了通用审查流水线（规则引擎 + MaxKB），没有针对"从图片/表格中识别出的文字"的专项审查逻辑 |
| 5 | **报告生成仅限 Excel** | 🟡 中 | 当前 `exportTaskReport()` 只能导出 Excel 格式。多模态识别场景通常需要 PDF 格式的图文混合报告 |
| 6 | **OCR API 密钥硬编码** | 🔴 高 | `OcrService` 的默认 API 密钥直接写在代码中（`sk-oilujlprpzdbnphjaqcbvktfwvskabjascfwolsezpuxysmy`），虽然会从数据库读取覆盖，但默认值不应该包含真实密钥 |
| 7 | **大文件 OCR 超时风险** | 🟡 中 | OCR 超时 180 秒，对于大型 PDF（50+ 页扫描件），base64 编码后的数据量可能触发 API 限制或超时 |

#### 3.5.3 改进建议

1. **表格结构化输出**：Python 微服务返回表格的行列结构（而非纯文本），前端可还原展示
2. **引入公式识别模型**：集成 LaTeX-OCR 或 Mathpix API，将公式转为 LaTeX/MathML 格式
3. **多模态专项审查**：对表格数据做数值一致性检查，对公式做计算验证
4. **PDF 报告生成**：支持生成图文混合的 PDF 审查报告
5. **OCR API 密钥安全**：移除硬编码默认密钥，强制从数据库读取
6. **分页 OCR 处理**：对大型 PDF 按页发送 OCR 请求，避免超时

### 4.0.5 修复方向

```typescript
export async function runAllRules(ctx: FileContext): Promise<RuleIssue[]> {
  // 1. 从数据库读取启用的规则
  const dbRules = await prisma.reviewRule.findMany({ where: { enabled: true } });
  const enabledCodes = new Set(dbRules.map(r => r.ruleCode));
  const severityMap = new Map(dbRules.map(r => [r.ruleCode, r.severity]));
  const configMap = new Map(dbRules.map(r => [r.ruleCode, r.config as any]));

  // 2. 按需执行规则（仅执行 enabled 的规则类别）
  const issues: RuleIssue[] = [];
  if (hasEnabledCategory(enabledCodes, 'NAME')) issues.push(...checkNaming(ctx, configMap));
  if (hasEnabledCategory(enabledCodes, 'CODE')) issues.push(...checkEncodingConsistency(ctx, configMap));
  // ...

  // 3. 覆盖 severity 为用户配置值
  return issues.map(i => ({
    ...i,
    severity: severityMap.get(i.ruleCode) || i.severity,
  }));
}
```

同时需要修改各规则函数，接受 `config` 参数覆盖硬编码的正则和阈值。

## 六、总结

当前系统的核心架构问题是 **"单一入口 + 全量流水线"**，导致六种审查模式的逻辑交叉缠绕在一起。具体表现为：

1. **以库审文**：实现最完善，但 MaxKB 依赖过强、审查结果缺少标准条文溯源
2. **以文审文**：完全未实现，缺少上游参照文件模型和比对引擎
3. **全文一致性**：仅实现格式层面的一致性（2条规则），语义层面的矛盾检测和跨文件一致性完全缺失
4. **错别字/语法**：规则引擎的 typo 模块名不副实，LLM 错别字检查不可控且启动成本过高
5. **多模态识别**：OCR 仅作为文本提取降级方案使用，表格/公式识别和结构化审查完全未实现
6. **自定义规则审查**：已有前端管理页面和数据库表，但审查流程从未读取用户配置 — **这是功能性 Bug，用户的所有规则自定义操作（启用/禁用/修改严重度/修改参数）均不生效**

### 自定义规则审查的定位

自定义规则审查在六大模式中的定位是 **"横切配置层 + 独立入口"**：

```
                    ┌─────────────────────────────────────┐
                    │         自定义规则配置层              │
                    │   (review_rules 表: enabled/         │
                    │    severity/config)                  │
                    └──────────────┬──────────────────────┘
                                   │ 被所有模式读取
          ┌────────────┬───────────┼───────────┬────────────┐
          ▼            ▼           ▼           ▼            ▼
    ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐
    │ 以库审文  │ │ 以文审文  │ │ 全文一致性 │ │ 错别字语法│ │ 多模态识别 │ │自定义规则 │
    │          │ │          │ │          │ │          │ │          │ │  (独立)   │
    │ 规则引擎  │ │ 规则引擎  │ │ 规则引擎  │ │ 规则引擎  │ │ 无规则   │ │ 仅规则引擎│
    │ +MaxKB   │ │ +MaxKB   │ │ +MaxKB   │ │ +LLM    │ │ +OCR    │ │ 不调AI   │
    └──────────┘ └──────────┘ └──────────┘ └──────────┘ └──────────┘ └──────────┘
```

- **作为独立入口**：`CUSTOM_RULE_REVIEW` 模式只跑规则引擎，不调 AI，速度最快、成本为零
- **作为配置层**：其他 5 种模式在执行规则引擎步骤时，也读取 `review_rules` 表的用户配置
- **修复优先级最高**：当前数据库与执行引擎的断裂是 Bug 级问题，必须优先修复

**推荐优先实施 P0 阶段**（修复规则配置断裂 + Schema 扩展 + 流水线策略模式 + 前端工作台），从根本上解耦审查模式，为后续各模式的独立迭代奠定基础。

> **✅ P0/P1 架构解耦已完成**（2026-04-15），详见第七章。
