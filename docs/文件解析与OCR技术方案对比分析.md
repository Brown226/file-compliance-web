## 二、各方案深度分析

### 2.1 markitdown（Microsoft）

**技术原理**：
- 针对每种文件格式使用专用 Python 库进行转换
- DOCX → `python-docx` 读取段落样式/表格 → 映射为 Markdown 语法
- XLSX → `openpyxl` 逐行读取 → Markdown 表格 `| col | col |`
- PDF → `pdfminer.six` 文本提取 + 表格识别 → Markdown
- PPTX → `python-pptx` 逐幻灯片 → `## 幻灯片 N`
- 图片 → PIL 提取 EXIF + LLM 视觉描述
- 输出为纯 Markdown 字符串，天然适合 LLM 消费

**优势**：
| 维度 | 说明 |
|------|------|
| LLM 天然友好 | 输出 Markdown，LLM 原生理解标题/表格/列表结构 |
| 格式覆盖广 | 29+ 格式（DOCX/XLSX/PDF/PPTX/HTML/图片/音频/EPub/邮件） |
| 轻量级 | 纯 Python 库，无 GPU 依赖，pip install 即用 |
| 插件生态 | 支持 OCR 插件（markitdown-ocr）、MCP 服务器 |
| 微软背书 | AutoGen 团队维护，112k Stars，生态成熟 |
| Token 高效 | Markdown 比 JSON 结构节省 30-50% Token |

**劣势**：
| 维度 | 说明 |
|------|------|
| PDF 表格精度 | 基于 pdfminer，中文复杂表格识别不如 MinerU/Docling |
| 无布局分析 | 不做视觉布局检测，对扫描 PDF 效果差 |
| 不支持 DWG | 当前系统独有需求 |
| 无 OCR 内置 | 需额外插件 + LLM API 调用 |

---

### 2.2 Docling（IBM Research）

**技术原理**：
- 基于 AI 视觉模型的专业 PDF 解析管线
- **布局分析**：使用 DocLayNet 模型（IBM 自研），检测标题/段落/表格/图片/页眉页脚等区域
- **表格结构识别**：使用 TableFormer 模型，识别合并单元格/跨行跨列
- **阅读顺序还原**：多栏布局的阅读顺序智能推断
- 输出为 DoclingDocument 格式（JSON），可导出 Markdown

**优势**：
| 维度 | 说明 |
|------|------|
| PDF 解析精度最高 | AI 视觉模型驱动，复杂布局/表格识别强 |
| 结构化输出 | DoclingDocument JSON，保留完整文档语义树 |
| 表格识别强 | TableFormer 模型，合并单元格/跨页表格 |
| 公式识别 | 支持 LaTeX 公式提取 |
| IBM 背书 | 技术报告 arXiv:2408.09869，持续迭代 |

**劣势**：
| 维度 | 说明 |
|------|------|
| GPU 依赖 | DocLayNet + TableFormer 需 GPU 推理，CPU 模式极慢 |
| 部署重 | 依赖 PyTorch + 深度学习模型，Docker 镜像大 |
| 中文支持 | 基础可用但不如 MinerU 专门优化 |
| 格式覆盖 | 主要聚焦 PDF，DOCX/XLSX 支持较弱 |
| 不支持 DWG | 与当前系统 DWG 需求不兼容 |

---

### 2.3 MinerU（上海AI实验室/OpenDataLab）

**技术原理**：
- 专为中文文档设计的 PDF 智能提取工具
- 内置版面分析模型，检测标题/正文/表格/图片/公式区域
- 表格智能还原：攻克旋转、跨页、合并单元格
- 公式精准转化：输出 LaTeX/MathML
- OCR 内置：多语言 OCR（含中文竖排、手写体）

**优势**：
| 维度 | 说明 |
|------|------|
| 中文 PDF 最佳 | 专为中文优化，表格/公式/竖排文字识别 SOTA |
| 58.5k Stars | 开源社区活跃，中文生态最完善 |
| 三端部署 | 在线 API + 离线部署 + 桌面客户端 |
| MCP 协议 | 原生支持 MCP，可对接 Agent 框架 |
| 多格式输出 | Markdown / JSON / LaTeX / CSV / HTML |
| 化学论文解析 | 分子检测、化学反应提取（独特能力） |

**劣势**：
| 维度 | 说明 |
|------|------|
| 聚焦 PDF | 不支持 DOCX/XLSX/PPTX 直接解析 |
| GPU 推荐 | 离线部署推荐 GPU，CPU 模式较慢 |
| 不支持 DWG | 与当前系统 DWG 需求不兼容 |
| 依赖较重 | 依赖 PyTorch + 多个 AI 模型 |

---

### 2.4 Marker（Vik Paruchuri）

**技术原理**：
- 基于 Nougat/Donut 等视觉模型的 PDF→Markdown 转换
- 端到端模型推理，不需要 OCR 引擎
- 支持表格、公式、代码块识别

**优势**：
| 维度 | 说明 |
|------|------|
| 速度较快 | 端到端模型，比 Docling/MinerU 快 |
| Markdown 输出 | 直接输出 Markdown |
| 无需 OCR 引擎 | 模型内置文字识别 |

**劣势**：
| 维度 | 说明 |
|------|------|
| GPL-3.0 许可 | 商业使用受限（当前系统为企业内部部署可接受） |
| 精度不如 Docling/MinerU | 复杂表格/多栏布局识别较弱 |
| GPU 必须 | 需 GPU 推理 |
| 中文支持一般 | 主要针对英文论文优化 |

---

### 2.5 Unstructured（Unstructured.io）

**技术原理**：
- 通用文档 ETL 平台，支持分区（partitioning）策略
- 检测文档元素类型（标题/正文/表格/列表/图片）
- 输出结构化元素列表

**优势**：
| 维度 | 说明 |
|------|------|
| 格式覆盖最广 | PDF/DOCX/PPTX/HTML/图片/邮件等全格式 |
| 元素级分区 | 精确到每个文档元素的类型和坐标 |
| RAG 优化 | 原生支持分块策略（chunking） |
| 企业级 | 商业版有 SLA、SOC 2、HIPAA |
| 连接器生态 | Databricks/Elasticsearch/Google Drive 等 |

**劣势**：
| 维度 | 说明 |
|------|------|
| 精度中等 | 自动化管线强，但单格式精度不如专精工具 |
| 部署复杂 | 平台级产品，学习曲线陡峭 |
| 商业版贵 | 开源版功能有限，高级功能需付费 |
| 不支持 DWG | |

---

### 2.6 LlamaParse（LlamaIndex）

**技术原理**：
- LlamaIndex 推出的 SaaS 文档解析服务
- 支持多格式解析，输出 Markdown/结构化 JSON
- 深度集成 LlamaIndex RAG 框架

**优势**：
| 维度 | 说明 |
|------|------|
| 精度高 | 复杂布局/表格识别效果好 |
| RAG 原生集成 | LlamaIndex 生态无缝对接 |
| 无需部署 | SaaS 服务，API 调用即可 |
| 多格式 | PDF/DOCX/PPTX/HTML 等 |

**劣势**：
| 维度 | 说明 |
|------|------|
| SaaS 依赖 | 必须联网，内网部署不可用 |
| 数据安全 | 文档上传到第三方服务器 |
| 收费 | 按页数计费，大批量成本高 |
| 不支持 DWG | |

## 四、OCR 服务对比分析

### 4.1 当前方案：PaddleOCR-VL-1.5（硅基流动托管）

| 属性 | 值 |
|------|-----|
| 模型 | PaddlePaddle/PaddleOCR-VL-1.5 |
| 参数量 | 0.9B（超轻量） |
| API 格式 | OpenAI Chat Completions 兼容 |
| 调用方式 | 文件 → base64 → 多模态消息 |
| 语言支持 | 109 种语言 |
| 超时 | 180s |
| 托管方 | 硅基流动（SiliconFlow） |

**优势**：
- OpenAI 兼容 API，集成简单
- 0.9B 参数，推理速度快
- 109 种语言，中文优化
- 表格提取能力强（输出 HTML 标签）
- 公式识别准确
- 手写体识别支持

**劣势**：
- 依赖第三方云服务（硅基流动）
- 大文件 base64 编码传输效率低
- 无法完全本地部署（内网部署受限）
- 扫描版 PDF 精度不如 MinerU 内置 OCR

## 五、推荐方案

### 5.1 文件解析：markitdown + MinerU 混合方案

基于当前系统的内网部署要求和 DWG 特殊需求，推荐**混合方案**：

```
┌─────────────────────────────────────────────────────────────┐
│                    文件解析引擎选择策略                        │
│                                                              │
│  DOCX/XLSX/PPTX/HTML  →  markitdown（轻量、结构保留好）       │
│                                                              │
│  PDF（文字型）         →  markitdown（快速、结构保留）          │
│                                                              │
│  PDF（扫描型/复杂表格） →  MinerU（中文 SOTA、表格精准）       │
│                                                              │
│  DWG                  →  保留现有 Python 微服务（独有能力）    │
│                                                              │
│  OCR 降级             →  PaddleOCR-VL 本地部署（替代云服务）   │
└─────────────────────────────────────────────────────────────┘
```

### 5.2 OCR：PaddleOCR-VL 本地部署

推荐将当前硅基流动托管的 PaddleOCR-VL 改为**本地 vLLM 部署**：

| 改造项 | 说明 |
|--------|------|
| 部署方式 | 内网 GPU 服务器 + vLLM 推理服务 |
| API 兼容 | vLLM 原生支持 OpenAI 兼容 API |
| 零代码改动 | Node.js 端只需修改 API 地址配置 |
| 精度提升 | 可升级到 PaddleOCR-VL 最新版 |
| 数据安全 | 文件不离开内网 |

### 5.3 实施路线图

| 阶段 | 内容 | 风险 | 预期收益 |
|------|------|------|---------|
| **Phase 1** | Python 微服务集成 markitdown（DOCX/XLSX） | 低 | 文档结构保留，LLM 审查精度 +20% |
| **Phase 2** | PDF 解析升级（markitdown 文字型 + MinerU 扫描型） | 中 | PDF 表格识别 +50%，公式提取 |
| **Phase 3** | PaddleOCR-VL 本地部署（vLLM） | 低 | 内网完全自治，数据安全 |
| **Phase 4** | PPTX 支持（markitdown） | 低 | 新增文件类型 |
| **Phase 5** | 前端 Markdown 预览联动 | 低 | 原文预览体验大幅提升 |

### 5.4 不推荐方案及原因

| 方案 | 不推荐原因 |
|------|-----------|
| LlamaParse | SaaS 依赖，内网不可用，数据安全风险 |
| Marker | GPL-3.0 许可证争议，中文支持弱 |
| Unstructured | 过重，精度不如专精工具，学习成本高 |
| DeepSeek-OCR | 参数量大（2B），中文复杂场景不如 PaddleOCR-VL |
| Azure/Google Doc AI | 无法内网部署，按页计费成本高 |
