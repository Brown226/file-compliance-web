# 知识库架构二次优化：三大组件定位与协作

## 一、现状问题

当前系统中存在三套"标准相关"的数据管理机制，职责重叠、边界模糊：

| 组件 | 当前定位 | 问题 |
|------|---------|------|
| **标准库管理** | 存标准全文(content) + 元数据 + 同步到 MaxKB | content 大字段冗余（MaxKB 已有）；同步逻辑易出状态不一致 |
| **MaxKB 知识库** | RAG 审查的知识来源 | 用户要绕道本地才能上传文档；分段调整需跳转 MaxKB 平台 |
| **临时标准库** | 内存 Map，按用户隔离，补充比对范围 | 重启丢失；3 字段太少无法表达复杂引用；与标准库分离导致逻辑复杂 |

核心矛盾：**标准库试图同时承担"内容管理"和"索引校验"两个完全不同的职责**，而 MaxKB 才是内容管理的主场。

### ⚠️ 现状中的严重缺陷：审查结果无溯源

当前 MaxKB 审查流程中存在一个关键问题：**MaxKB 返回的参考原文（source 字段）被完全丢弃了。**

```typescript
// maxkb.service.ts:619-686 — reviewViaMaxKB() 返回值
interface MaxKBChatResult {
  content: string;        // ← 只用了这个（LLM 审查结果文本）
  source?: Array<{        // ← 这个被丢弃了！
    content: string;      //    知识库中的匹配段落原文
    document_name: string;//    来源文档名
    similarity: number;   //    相似度
  }>;
}

// base-pipeline.ts:118-119 — 调用方只取 content
const result = await MaxKBService.reviewViaMaxKB(...);
const parsed = LlmService.parseReviewResult(result.content);  // source 丢失！
```

**后果**：用户看到"违规"结果，但不知道依据的是哪条标准、哪段原文，无法确认审查结论的可信度。

## 二、重新定位

### 2.1 总体原则

> **让每个组件只做它最擅长的事，数据流单向，职责不重叠，审查结果可溯源。**

```
┌──────────────────────────────────────────────────────────────┐
│                                                              │
│   MaxKB 知识库                标准索引库               临时标准库  │
│   (知识内容层)                (业务索引层)             (比对补充层) │
│                                                              │
│   ┌──────────────┐      ┌──────────────┐     ┌──────────┐   │
│   │ 上传文档      │      │ 标准编号索引  │     │ 临时导入  │   │
│   │ 调整分段      │      │ 标准名称      │     │ 快速补充  │   │
│   │ 预览分块      │      │ 标识符        │     │ 一次性    │   │
│   │ Embedding    │      │ 实施状态      │     │ 用完即清  │   │
│   │ RAG 检索      │      │ 关联 MaxKB ID │     │          │   │
│   │              │      │              │     │          │   │
│   │ 回答: 内容    │      │ 回答: 是否存在 │     │ 回答:   │   │
│   │  该标准说了啥  │      │  该标准有没有  │     │  还有没  │   │
│   └──────────────┘      └──────────────┘     └──────────┘   │
│         │                      │                    │        │
│         │    RAG 审查           │  标准引用           │  补充   │
│         │    (LLM 对话)        │  规范性检查          │  比对   │
│         │                      │  (纯规则匹配)       │  范围   │
│         ▼                      ▼                    ▼        │
│   ┌──────────────────────────────────────────────────────┐   │
│   │                  审查引擎 (Review Pipeline)            │   │
│   │                                                      │   │
│   │  MaxKB 审查: 待审文本 → RAG 检索 → LLM → 审查结果     │   │
│   │                            ↓                         │   │

│   │                                                      │   │
│   │  引用检查:   《GB/T 12459-2017》→ 8级比对 → 是否有效  │   │
│   └──────────────────────────────────────────────────────┘   │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 2.2 三大组件定位详解

#### 🔵 MaxKB 知识库 — 知识内容层

| 维度 | 说明 |
|------|------|
| **核心职责** | 管理标准/规范文档的**全文内容**，提供 RAG 检索能力 |
| **数据内容** | 文档原文、分段结果、Embedding 向量 |
| **用户操作** | 上传文档、调整分段策略、预览/编辑分段、删除/更新文档 |
| **审查角色** | LLM 审查时的**知识来源** — "这个标准具体说了什么" |
| **溯源角色** | 审查结果中展示**参考原文段落** — "审查结论依据了哪段标准条文" |
| **不适合做** | 标准编号的有效性校验（MaxKB 是向量检索，不是精确匹配） |

**关键认知：MaxKB 里存的不是"标准编号列表"，而是"标准文档全文"。** 
一篇 `GB/T 8163-2018` 的文档在 MaxKB 里会被分成若干段落，用于 RAG 语义检索。
当 LLM 审查"无缝钢管规格是否符合要求"时，MaxKB 检索出相关段落作为上下文。

**知识库内容不限于有标准号的文档。** 可以是：
- 国家/行业标准：`GB/T 8163-2018`、`DL/T 935-2020`
- 企业内部规范：如 `QS-CNPE-XX-001`，可能没有标准号
- 法律法规：如《安全生产法》、《环境保护法》
- 技术规格说明：如通风管道规格说明、电缆选型手册
- 项目专用标准：如 `15251CI-JPS201-009` 这类项目编号文档

#### 🟢 标准索引库 — 业务索引层

| 维度 | 说明 |
|------|------|
| **核心职责** | 维护标准规范的**元数据索引**，供精确匹配和业务关联 |
| **数据内容** | 标准编号、标准名称、标识符、实施状态、废止日期、关联的 MaxKB 文档 ID |
| **用户操作** | 查看标准列表、搜索标准、关联/取消关联 MaxKB 文档、标记废止状态 |
| **审查角色** | 标准引用规范性检查的**比对基准** — "这个标准编号是否有效" |
| **不存储** | 标准全文内容（content 字段废弃，全文在 MaxKB 管理） |

**关键认知：标准索引库回答的是"有没有"的问题，不是"说了啥"的问题。**

**注意：不是所有知识库文档都需要在标准索引库中有对应记录。** 
- 有标准编号的文档（GB、DL、HJ 等）→ 需要在索引库中登记，用于引用规范性检查
- 无标准编号的文档（法律条文、规格说明、项目规范等）→ 只在 MaxKB 知识库中存在，通过 RAG 审查时自动检索，无需索引库登记

举例：待审文件引用了 `DL/T 935-2020`：
- 标准索引库查到 → 该标准有效，引用规范 ✅
- 标准索引库查到但状态为 ABOLISHED → 标准已废止，引用不规范 ❌
- 标准索引库查不到 → 需要进一步查临时标准库

#### 🟡 临时标准库 — 比对补充层

| 维度 | 说明 |
|------|------|
| **核心职责** | 补充标准索引库未覆盖的标准引用，降低误报率 |
| **数据内容** | 原始文本、提取的标准编号、标识符、来源文件、导入用户 |
| **用户操作** | 上传 Excel/手动添加、查看条目、比对测试、清理/归档 |
| **审查角色** | 标准引用检查的**补充白名单** — "这个标准虽然索引库没有，但临时库有，不算违规" |
| **生命周期** | 临时性质 — 按用户/任务隔离，可归档为正式标准，也可用完清理 |

**关键认知：临时标准库不是"知识库的临时版"，而是"比对白名单的补充"。**

为什么需要？因为：
1. **标准索引库永远不可能 100% 完备** — 新标准不断发布，内部标准不公开
2. **行业特殊性** — 如 `H44F46-10 GB 12235/GB 2236` 这类阀门型号+标准混写
3. **一次性审查场景** — 对照特定项目标准清单审查，无需正式入库

典型场景：用户上传的设备材料表（如 HX1CI Excel），第 8 列包含标准引用（GB/T 12459-2017、DL/T 935-2020 等）。如果索引库只覆盖了 30 个标准但文件引用了 50 个，剩余 20 个会全部报 VIOLATION（误报）。用户通过临时标准库导入该项目的标准清单，把误报率降到真实水平。

### 2.3 三者的数据流关系

```
用户上传标准文档 ──────────────────────────────▶ MaxKB 知识库
                                                      │
                                          自动/手动关联
                                                      │
                                                      ▼
                                           标准索引库新增一条记录
                                           (编号/名称/状态/maxkbDocId)
                                           [仅限有标准编号的文档]

用户上传临时标准清单 (Excel) ──────────────▶ 临时标准库
                                           (编号/名称/标识符/来源)

审查任务执行:
  ┌─ LLM 审查 ──▶ MaxKB.reviewViaMaxKB()
  │                    │
  │                    ├─ content → LLM 审查结果 (issueType/description/...)
  │                    └─ source → 参考原文段落 ← 必须保留并展示给用户！
  │                                   │
  │                                   ├─ document_name: 来源文档名
  │                                   ├─ content: 匹配的标准条文原文
  │                                   └─ similarity: 相似度
  │
  └─ 引用检查 ──▶ checkStandardReferences()
                    │
                    ├─ 比对标准索引库 (主力)
                    ├─ 比对临时标准库 (补充)
                    └─ 都没匹配上 ──▶ VIOLATION: "未找到该标准规范"
```

## 五、审查流程对照

### 5.1 改造前

```
LLM 审查流程:
  1. MaxKB.reviewViaMaxKB() → { content, source }
  2. parseReviewResult(result.content) → issues[]
  3. source 丢弃 ❌
  4. issues 存入 task_details

引用检查流程:
  1. Standard.findMany() → 获取标准库所有编号/名称/标识符
  2. StandardService.getTempLibrary(userId) → 从内存 Map 取临时库
  3. 两者合并为 checkLibrary
  4. 8 级比对 → 不匹配则报 VIOLATION
```

问题：source 丢弃导致无溯源；临时库重启丢失；标准库 content 冗余；同步状态难维护

### 5.2 改造后

```
LLM 审查流程:
  1. MaxKB.reviewViaMaxKB() → { content, source }
  2. parseReviewResult(result.content) → issues[]
  3. 将 source 附加到每个 issue 的 sourceReferences ✅
  4. issues + sourceReferences 一起存入 task_details

引用检查流程:
  1. Standard.findMany({ where: { standardNo: { not: null } } })
     → 标准索引库（含 OFFICIAL + TEMP_ARCHIVED 来源）
  2. TempStandardEntry.findMany({ where: { userId, taskId } })
     → 临时标准库（持久化，按用户+任务隔离）
  3. 两者合并为 checkLibrary
  4. 8 级比对 → 不匹配则报 VIOLATION
```

收益：source 保留可溯源；临时库不丢失；标准库无冗余 content；不再需要同步状态追踪

## 七、设计反思与不足

### 当前方案的不足

| 不足 | 说明 | 后续优化方向 |
|------|------|-------------|
| **source 与 issue 的对应关系模糊** | MaxKB 返回的 source 是整个对话的参考段落，不是逐个 issue 的。一个 chunk 可能产生 3 个 issue，但 source 是共享的 | 需要在 prompt 中要求 LLM 在每个 issue 中标注引用的 source 序号，或通过分段审查（每个 chunk 独立）实现精确映射 |
| **标准索引库需要手动维护** | MaxKB 上传新文档后，标准索引库不会自动新增记录 | 可通过 MaxKB Webhook 回调自动同步；或在审查前自动从 MaxKB 拉取文档列表与索引库比对 |
| **临时标准库与标准索引库可能重叠** | 用户可能把已存在的标准又导入临时库 | 导入时自动去重检查，已存在的跳过 |
| **MaxKB 依赖** | MaxKB 不可用时 LLM 审查完全失败 | 需要保留 LLM 直连降级方案（当前已有 `llm_only` 模式） |
| **分段映射粒度** | source 只是段落片段，不是精确到条文号 | 可通过 prompt 要求 LLM 返回 `standardRef` 字段标注条文号，结合 source 原文双重溯源 |

### 长期演进方向

1. **智能索引同步** — MaxKB 上传文档后自动提取标准编号，写入标准索引库
2. **条文级溯源** — source 段落 + LLM 标注的条文号 → 精确到"第X条第Y款"的溯源
3. **跨标准冲突检测** — 引用的多个标准之间存在矛盾时告警
4. **标准版本追踪** — 标准更新时自动标记使用旧版的审查任务

## 八、核心收益总结

| 收益 | 说明 |
|------|------|
| **职责清晰** | MaxKB 管内容，标准索引管元数据，临时库管补充比对 |
| **消除冗余** | content 大字段去掉，同步逻辑去掉 |
| **降低误报** | 临时标准库持久化，用户不再因重启丢失比对白名单 |
| **数据一致** | MaxKB 是知识内容唯一真相源，标准索引只是引用 |
| **用户体验** | 在 MaxKB 调分段一步到位，无需同步等待 |
| **审查可溯源** | source 参考原文保留并展示，用户可确认审查结论依据 |
