# 文件智能审查系统 — 架构设计与功能规格

> 版本：v1.0 | 更新日期：2026-04-14

---

## 一、系统概述

**文件智能审查系统**是一个面向核电工程领域的企业级 Web 应用，核心能力是将上传的工程文档（DWG/Word/Excel/PDF）与企业合规标准进行自动比对，通过**确定性规则引擎 + MaxKB RAG 知识库 + LLM** 三层审查机制，输出结构化的违规问题清单。

### 定位

- **MVP 阶段**：Node.js/Express 单体架构，内网部署，MaxKB 作为唯一 AI 引擎
- **生产目标**：微服务架构（Python/FastAPI + Celery + MinIO + pgvector），见架构设计文档

---

## 二、技术栈

| 层次 | 技术选型 | 说明 |
|------|----------|------|
| **前端** | Vue 3 + Vite 6 + Element Plus + ECharts + Pinia + Vue Router | Composition API，Pinia 状态持久化 |
| **后端** | Node.js + Express 5 + TypeScript (CommonJS) | 静态方法服务类，Prisma ORM |
| **数据库** | PostgreSQL 15 | Prisma 5，`db push` 开发模式 |
| **缓存** | Redis 7 | JWT 黑名单、MaxKB Token 缓存 |
| **AI 引擎** | MaxKB 2.8.0（内网部署） | 知识库管理 + RAG 检索 + LLM 对话 |
| **OCR** | 硅基流动 PaddleOCR-VL-1.5 | OpenAI 兼容接口，base64 传输 |
| **文件解析** | Python FastAPI 微服务 (PyMuPDF + pdfplumber + python-docx + openpyxl + ezdxf) | 结构化解析（段落/表格/样式/DWG图层+标注） |
| **容器化** | Docker Compose | PostgreSQL + Redis |

### 关键模型字段

| 模型 | 核心字段 | 说明 |
|------|----------|------|
| **User** | username, passwordHash, name, role(ADMIN/MANAGER/USER), departmentId | 三级 RBAC |
| **Standard** | title, content, version, isActive, folderId, maxkbSyncStatus, maxkbDocumentId, maxkbKnowledgeId | 含 MaxKB 同步状态 |
| **StandardFolder** | name, parentId (自引用) | 树形分类 |
| **Task** | title, description, status(PENDING/PROCESSING/COMPLETED/FAILED), creatorId | 审查任务 |
| **TaskFile** | fileName, filePath, fileType, errorCount | 关联文件 |
| **TaskDetail** | issueType, ruleCode, severity, originalText, suggestedText, description, cadHandleId | 审查结果 |
| **SystemConfig** | key(unique), value(JSON) | 运行时配置（LLM/OCR/MaxKB） |
| **ReviewRule** | ruleCode(unique), category, severity, enabled, config(JSON) | 确定性规则配置 |

## 六、核心业务流程

### 6.1 文件审查流水线

```
用户上传文件 → TaskService.createTask()
                     │
                     ▼ setImmediate() (非阻塞)
            ReviewService.processTask()
                     │
            ┌────────▼────────────────────────────────┐
            │         遍历每个 TaskFile                │
            │                                          │
            │  1. ParserService.parseFile()            │
            │     ├─ docx → mammoth 提取纯文本         │
            │     ├─ xlsx → XLSX 提取单元格            │
            │     ├─ pdf  → pdf-parse 提取文本          │
            │     └─ dwg  → 占位，返回空字符串          │
            │                                          │
            │  2. OCR 降级（文本为空时）                │
            │     └─ OcrService.recognizeFile()        │
            │        → PaddleOCR-VL base64 识别        │
            │                                          │
            │  3. PDF 逐页提取（页眉/页码规则需要）     │
            │     └─ ParserService.parsePdfPages()     │
            │                                          │
            │  4. RuleEngineService.runAllRules()      │
            │     10 大类确定性规则（不依赖 LLM）       │
            │     ├─ NAME_001~010 文件命名              │
            │     ├─ CODE_001~005 编码一致性            │
            │     ├─ ATTR_001~011 封面属性              │
            │     ├─ HEADER_001~003 页眉                │
            │     ├─ PAGE_001~004 页码                  │
            │     ├─ FORMAT_001~005 格式规范            │
            │     ├─ COMPL_001~003 数据完整性           │
            │     ├─ CONSIST_001~002 一致性             │
            │     ├─ LAYOUT_001 排版布局                │
            │     └─ TYPO_001 正文编码                  │
            │                                          │
            │  5. MaxKB AI 审查（唯一 AI 引擎）         │
            │     ├─ 健康检查                           │
            │     ├─ LlmService.splitText(4000) 分片   │
            │     └─ MaxKBService.reviewViaMaxKB()      │
            │        → POST /chat/api/{app}/completions │
            │        → RAG 检索 + LLM 生成审查结果      │
            │                                          │
            │  6. 结果写入                              │
            │     ├─ TaskDetail.createMany()            │
            │     └─ TaskFile.errorCount 更新           │
            └──────────────────────────────────────────┘
                     │
                     ▼
            Task.status → COMPLETED / FAILED
```

### 6.2 MaxKB 集成初始化流程

```
管理员点击「一键初始化」

         │
         ├─ 1. getDefaultWorkspaceId()

         │

         │        ├─ getEmbeddingModels() ← GET /model_list?model_type=EMBEDDING

         │
         ├─ 3. ensureReviewApplication(workspaceId, knowledgeId)
         │     ├─ listApplications() → 查找「文件合规审查」

### 6.4 Word 文件审查全流程详解（端到端）

以用户上传一个 Word 文件为例，完整追踪从前端操作到结果展示的全过程：

```
用户上传 "XX工程设计报告.docx"
        │
        ▼
┌─────────────────────────────────────────────────────────────────────┐
│ 1. 前端：创建审查任务                                                │
│    NewTask.vue → POST /api/tasks (multipart/form-data)              │
│    携带：title, description, standardIds[], files[]                  │
│    文件通过 multer 上传到 backend/uploads/                            │
└─────────────────────────────────────────────────────────────────────┘
        │
        ▼
┌─────────────────────────────────────────────────────────────────────┐
│ 2. 后端：TaskService.createTask()                                   │
│    ├── 创建 Task 记录 (status = PROCESSING)                          │
│    ├── 创建 TaskFile 记录 (fileName, filePath, fileType, fileSize)   │
│    ├── 创建 TaskStandard 多对多关联                                   │
│    └── setImmediate() → ReviewService.processTask() 异步触发         │
│        （不阻塞 HTTP 响应，立即返回任务信息给前端）                      │
└─────────────────────────────────────────────────────────────────────┘
        │
        ▼
┌─────────────────────────────────────────────────────────────────────┐
│ 3. 审查引擎：ReviewService.processFile() — 逐文件处理                 │
│                                                                     │
│  Step 1: 文件解析 ParserService.parseFile("docx")                   │
│    └── mammoth.extractRawText() → 纯文本字符串                        │
│        例："1.项目概况\n本项目位于XX市...\n2.设计依据..."               │
│                                                                     │
│  Step 2: OCR 降级（Word 不需要，跳过）                                │
│    └── needsOcr() 仅 PDF 文本 < 20 字符 或 DWG 时触发                │
│                                                                     │
│  Step 3: PDF 逐页提取（非 PDF，跳过）                                 │
│    └── 仅 PDF 需要逐页文本用于页眉/页码规则                            │
│                                                                     │
│  Step 4: 规则引擎 RuleEngineService.runAllRules()                    │
│    ├── 命名规范 (NAME_001~010): 文件名是否符合企业命名规则             │
│    ├── 格式规范 (FORMAT_001~005): 全角半角混用、多余空格等             │
│    ├── 数据完整性 (COMPL_001~003): 关键字段是否缺失                    │
│    ├── 一致性 (CONSIST_001~002): 日期/编号格式是否全文一致             │
│    ├── 排版布局 (LAYOUT_001): 行首缩进、中英文间空格                   │
│    └── 正文编码 (TYPO_001): 确定性错别字检查                           │
│    → 输出 RuleIssue[]，写入 task_details 表                          │
│                                                                     │
│  Step 5: DWG 特殊处理（非 DWG，跳过）                                 │
│                                                                     │
│  Step 6: 无文本跳过检查（有文本，跳过）                                │
│                                                                     │
│  Step 7: MaxKB AI 审查                                               │
│    ├── 检查 MaxKB 是否初始化 (applicationId + apiKey)                 │
│    ├── 健康检查 MaxKBService.healthCheck()                            │
│    ├── 文本分片 LlmService.splitText(text, 4000)                     │
│    │   └── 超过 4000 字符按段落分割为多个 chunk                       │
│    ├── 逐片审查 MaxKBService.reviewViaMaxKB()                         │
│    │   └── POST /chat/api/{applicationId}/chat                       │
│    │       发送: { message: chunk, re_chat: true }                    │
│    │       MaxKB 内部: RAG 检索知识库标准 → LLM 生成审查意见            │
│    ├── 解析结果 LlmService.parseReviewResult()                        │
│    │   └── 期望 JSON: [{ issueType, originalText, suggestedText,      │
│    │                     description, cadHandleId }]                  │
│    └── 写入 task_details 表 (severity: TYPO→warning, VIOLATION→error) │
│                                                                     │
│  Step 8: 更新文件错误计数 updateFileErrorCount()                      │
└─────────────────────────────────────────────────────────────────────┘
        │
        ▼
┌─────────────────────────────────────────────────────────────────────┐
│ 4. 任务状态更新                                                      │
│    ├── 全部文件成功 → status = COMPLETED                              │
│    ├── 全部文件失败 → status = FAILED                                 │
│    └── 部分失败 → status = COMPLETED（仍标记为完成）                   │
└─────────────────────────────────────────────────────────────────────┘
        │
        ▼
┌─────────────────────────────────────────────────────────────────────┐
│ 5. 前端：任务详情页 TaskDetails.vue                                   │
│                                                                     │
│  布局:                                                               │
│  ┌──────────────────────────────────────────────────┐               │
│  │ 顶部信息栏: 标题 | 创建人 | 状态 | 时间             │               │
│  │ 统计条: 错误:3 警告:5 提示:2 | 命名:1 格式:4 ...   │               │
│  │ [导出审查报告]                                      │               │

#### 关键数据流总结

```
uploads/xx.docx
    ↓ mammoth.extractRawText()
纯文本
    ↓ RuleEngineService.runAllRules() (确定性规则，不依赖 LLM)
RuleIssue[] → task_details 表 (issueType, ruleCode, severity, originalText, suggestedText, description)
    ↓ LlmService.splitText(4000) (按段落分片，每片 ≤ 4000 字符)
chunk[]
    ↓ MaxKBService.reviewViaMaxKB() (逐片发送)
    │   POST /chat/api/{applicationId}/chat
    │   MaxKB 内部: 知识库 RAG 检索 + LLM 生成
ReviewIssue[] → task_details 表 (issueType, severity, originalText, suggestedText, description)
    ↓
前端轮询 GET /api/tasks/:id + GET /api/tasks/:id/details
    ↓
TaskDetails.vue 卡片式展示 (分类标签 + 严重度 + 原文本 + 建议修改)
```

#### 不同文件类型的审查差异

| 文件类型 | 解析方式 | OCR 降级 | 逐页提取 | 规则引擎适用项 | MaxKB 审查 |
|---------|---------|---------|---------|--------------|-----------|
| **docx** | mammoth 纯文本 | 不需要 | 不支持 | NAME + FORMAT + COMPL + CONSIST + LAYOUT + TYPO | ✅ |
| **xlsx/xls** | XLSX 单元格文本 | 不需要 | 不支持 | NAME + FORMAT + COMPL + CONSIST + TYPO | ✅ |
| **pdf (文字版)** | pdf-parse 文本 | 不需要（文本≥20字） | ✅ pdfjs-dist 逐页 | NAME + CODE + ATTR + HEADER + PAGE + FORMAT + COMPL + CONSIST + LAYOUT + TYPO | ✅ |
| **pdf (扫描件)** | pdf-parse 返回空 | ✅ PaddleOCR-VL | OCR 后逐页 | 同上（依赖 OCR 质量） | ✅（依赖 OCR 结果） |
| **dwg** | 占位空实现 | 需要（暂无 OCR 支持） | 不支持 | 仅 NAME | ❌（直接返回提示性违规） |

## 八、认证与权限

### 8.1 JWT 认证流程

```
1. POST /api/auth/login → 验证用户名密码 → 生成 JWT (payload: {id, role, departmentId})
2. 前端存入 Pinia (localStorage 持久化)
3. Axios 拦截器自动附加 Authorization: Bearer <token>
4. authenticate 中间件: 提取 Token → Redis 黑名单检查 → jwt.verify → 注入 req.user
5. 登出时: POST /api/auth/logout → Token 加入 Redis 黑名单
```

### 8.2 RBAC 三级权限

| 角色 | 数据范围 | 典型操作 |
|------|----------|----------|
| **ADMIN** | 全局 | 用户管理、标准CRUD、MaxKB初始化/同步、规则编辑 |
| **MANAGER** | 本部门 + 子部门 | 查看本部门任务、创建任务、查看标准 |
| **USER** | 仅自己 | 查看自己的任务、创建任务、查看标准 |

`getTaskFilterByRole()` 动态构建 Prisma `where` 条件，递归获取子部门 ID。

## 十、确定性规则引擎

`RuleEngineService` 实现 10 大类、30+ 条确定性审查规则，不依赖 LLM：

| 规则代码 | 分类 | 检查内容 | 严重度 |
|----------|------|----------|--------|
| NAME_001~010 | 命名规范 | 文件名格式、扩展名、编码字符、禁止特殊字符 | warning~error |
| CODE_001~005 | 编码一致性 | 封面编码与文件名一致、目录编码与正文匹配 | error |
| UNIT_001~005 | 编码一致性 | 计量单位统一性 | warning |
| ATTR_001~011 | 封面属性 | 必填字段检查（图册编号、密级、工程号等） | error |
| HEADER_001~003 | 页眉检查 | 页眉内容一致性、编码匹配 | warning |
| PAGE_001~004 | 页码检查 | 连续性、存在性、格式 | warning |
| FORMAT_001~005 | 格式规范 | 列表标记、引用格式、变更标记 | warning |
| COMPL_001~003 | 数据完整性 | 必填字段非空、电缆路径完整 | error |
| CONSIST_001~002 | 一致性 | 封面与正文信息一致 | warning |
| LAYOUT_001 | 排版布局 | 中英文混排空格 | info |
| TYPO_001 | 正文编码 | 常见错别字检测 | warning |

规则配置存储在 `ReviewRule` 表中，支持：
- 按规则代码启用/禁用
- 自定义严重度（error/warning/info）
- 高级配置（JSON：正则、阈值等）
