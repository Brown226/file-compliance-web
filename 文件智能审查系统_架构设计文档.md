# 文件智能审查系统（企业内网版）产品需求与架构设计说明书

## 1. 项目概述
### 1.1 项目背景与目标
本项目旨在将现有的「Windows 客户端 + CAD 插件」形态的文件智能审查工具，升级为**企业端内网私有化部署的网页版应用（Web App）**。在保留现有的“标准号引用有效性核查、常见错别字与语法错误核查、引用项目信息核查”等核心能力前提下，补齐企业级的高并发处理、数据资产治理、多租户/部门权限隔离及安全合规能力。

### 1.2 核心运行环境约束
*   **部署环境：** 纯内网离线环境（无公网访问权限）。
*   **算力支持：** 私有化 CPU 集群 + 固定的本地 GPU 推理节点。
*   **外部依赖：** 禁止调用任何公有云 API（包括云端大模型、云端 CAD 解析渲染服务等）。

## 2. 总体产品形态与业务边界
基于内网环境及研发成本考量，系统的业务边界确立如下：

1.  **图纸定位与交互策略（端云结合）：** 
    *   Web 端作为“任务控制台”与“报告看板”，仅展示问题列表与文本上下文。
    *   通过极简的 **“剪贴板 + CAD Lisp 脚本”** 方案实现 0 开发门槛的本地 CAD 图纸高亮定位闭环。
2.  **文件存储与安全策略（加密留档）：** 
    *   用户上传的原文件（DWG/Office等）统一存储至内网对象存储，实施服务端加密。
    *   支持历史任务追溯与未来大模型升级后的“一键复审”。
3.  **系统容量与性能基线：** 
    *   **典型任务规模：** 100 个文件 / 批次。
    *   **并发峰值要求：** 支撑 20 个项目（Job）同时排队/处理（即 2000 个文件并发入队）。
    *   **SLA 预期：** 典型批次任务（100 个文件）在 30 分钟内审查完毕。
4.  **审查结果流转深度（只读报告型）：** 
    *   系统定位为高效的“智能审查报告生成器”。
    *   结果在 Web 端支持多维度筛选与查看，最终导向 Excel 导出，不涉及复杂的系统内工单状态机流转。

## 3. 系统整体架构与技术栈选型

### 3.1 核心技术栈
*   **前端展示层：** Vue 3 + TypeScript + Element Plus / Ant Design Vue（支持大表格虚拟滚动）。
*   **后端 API 与调度层：** Python (FastAPI) —— 极佳的异步 I/O 性能，天生契合 AI 与数据处理生态。
*   **分布式任务队列：** Celery + RabbitMQ（消息中间件） + Redis（状态缓存与限流）。
*   **关系型数据库：** PostgreSQL + `pgvector` 插件（支持业务数据存储及标准库的向量化 RAG 检索）。
*   **对象存储引擎：** MinIO（支持 SSE-S3 服务端加密与预签名 URL 下载）。
*   **AI 推理与解析层：** 
    *   LLM 引擎：vLLM 推理框架 + Qwen3-32B 本地模型。
    *   CAD 解析：Python `ezdxf` 库或 ODA 离线 SDK。

### 3.2 物理部署拓扑（微服务化架构）
*   **Node 1 (网关与存储节点 - CPU)：** Nginx、MinIO、PostgreSQL、Redis、RabbitMQ。
*   **Node 2 (API 服务节点 - CPU)：** FastAPI Server（处理 HTTP 请求、鉴权、WebSocket 进度推送）。
*   **Node 3 (解析 Worker 节点 - CPU)：** 部署多个 Celery Worker 实例，专职从 MinIO 拉取图纸并进行 CPU 密集型的 DWG 解析与文本/Handle 提取。
*   **Node 4 (AI 推理节点 - GPU)：** 部署 vLLM 提供大模型 API；部署专属 Celery Worker 执行 Prompt 组装与 AI 审查编排逻辑。

## 4. 核心业务流程与数据流

### 4.1 异步审查主链路
1.  **分片上传：** 用户在 Web 端选择图纸/文档，前端分片直传至后端，后端合并后存入 MinIO。
2.  **任务拆分入队：** FastAPI 创建主任务记录（Job），将其拆分为 N 个子任务（Task，对应单个文件），推入 RabbitMQ 的 `parse_queue`。
3.  **并行解析提取：** Parse Worker 消费队列，拉取文件，解析出纯文本内容及对应的 **CAD Handle ID**，结构化数据存入 Redis/PostgreSQL。
4.  **AI 审查编排：** 解析完成的 Task 进入 `review_queue`。Review Worker 执行流水线：
    *   *规则初筛：* 正则表达式匹配基础格式。
    *   *向量检索：* 通过 `pgvector` 从标准库中检索相关规范条文。
    *   *LLM 判断：* 将文本、规范条文组装为 Prompt，请求本地 vLLM 接口进行误报过滤与语义审查。
5.  **结果持久化：** 审查出的问题条目（含 Handle ID）写入 PostgreSQL 的 Result 表。
6.  **通知与导出：** Job 下所有 Task 完成后，触发导出服务生成 Excel 报告，存入 MinIO，通过 WebSocket 通知前端“审查完成”。

## 5. 关键技术难点与解决方案

### 5.1 0门槛 CAD 定位闭环方案（Lisp 剪贴板法）
*   **服务端支撑：** 解析引擎提取文本时，强制绑定提取其对象句柄（Handle ID）。
*   **前端交互：** 用户点击结果列表的【定位】按钮，前端调用 `navigator.clipboard.writeText("Handle_ID")` 将 ID 写入剪贴板。
*   **CAD 端脚本（用户本地加载 `locate.lsp`）：**
    ```lisp
    (defun C:FND (/ hnd obj)
      (setq hnd (getstring "\n请输入要定位的 Handle ID (Ctrl+V 粘贴): "))
      (if (and hnd (/= hnd ""))
        (progn
          (setq obj (handent hnd))
          (if obj
            (progn
              (sssetfirst nil (ssadd obj))
              (command "_.ZOOM" "_Object" obj "")
              (princ (strcat "\n成功定位到 Handle: " hnd))
            )
            (princ (strcat "\n未找到对象。"))
          )
        )
      )
      (princ)
    )
    ```

### 5.2 大并发与 LLM 资源保护
*   **队列隔离：** 设置不同优先级的 RabbitMQ 队列，确保紧急项目优先处理。
*   **LLM 批处理（Continuous Batching）：** vLLM 框架开启批处理能力，合并多个短文本审查请求，最大化 GPU 吞吐率。
*   **限流熔断：** Celery 层面控制并发发往 vLLM 的请求数，防止 OOM（显存溢出）。

### 5.3 内网文件安全策略
*   **禁止直链：** Web 端获取下载链接时，API 实时向 MinIO 申请 **短期预签名 URL（Presigned URL）**，超时即失效。
*   **服务端加密：** MinIO 开启数据落盘加密，防止物理层面的数据泄露。

## 6. 数据库核心表结构设计

*   **`tenant_org_table` (租户/组织表)：** `id`, `name`, `parent_id` (支持部门层级)。
*   **`job_table` (主任务表)：** `job_id`, `creator_id`, `tenant_id`, `total_files`, `status` (PENDING/RUNNING/COMPLETED/FAILED), `created_at`。
*   **`task_table` (子任务表)：** `task_id`, `job_id`, `file_name`, `minio_path`, `status`, `parse_time`, `review_time`。
*   **`result_table` (审查结果表)：** 
    *   `id`, `task_id`, `error_type` (错别字/规范错误等)。
    *   `error_desc` (大模型给出的判定描述)。
    *   `original_text` (原图纸文本), `standard_ref` (违反的标准号)。
    *   `cad_handle_id` (用于 CAD 定位的关键标识)。
*   **`standard_db_table` (企业标准库)：** `id`, `standard_code`, `content`, `embedding` (由 pgvector 存储的向量表示), `version`, `is_active`。

## 7. 实施路线图 (MVP 到规模化落地)

### Phase 1: 核心链路打通 (MVP，预计 3-4 周)
*   搭建 FastAPI + PostgreSQL + MinIO + Redis 基础骨架。
*   实现 DWG 文件到 纯文本 + Handle 的离线解析提取。
*   完成 Qwen3-32B 的 vLLM 本地部署及 API 封装。
*   完成基础的 Web 上传、审查结果展示及 Lisp 脚本联调验证。

### Phase 2: 异步并发与性能提升 (预计 3 周)
*   全面引入 Celery + RabbitMQ，将解析与审查拆分为异步 Worker 任务。
*   实现 WebSocket 进度推送，支持 100 文件/批次的并发排队。
*   增加 API 限流及 LLM 侧并发保护机制。

### Phase 3: 企业标准库与 AI 深度融合 (预计 3 周)
*   开发标准库与项目字典的管理后台（CRUD）。
*   引入 `pgvector` 向量数据库能力，实现 RAG（检索增强生成），提升审查精准度。
*   开发 Excel 报告异步打包导出功能。

### Phase 4: 权限合规与生产加固 (预计 2 周)
*   接入企业 SSO（如 LDAP/AD）完成用户认证。
*   实现基于部门/项目的 RBAC 权限隔离。
*   全链路日志埋点、敏感信息脱敏与系统压力测试，交付上线。
